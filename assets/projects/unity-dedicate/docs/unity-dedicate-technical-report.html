01 · OVERVIEW

프로젝트 개요

Mirror 튜토리얼이 아닌, 실제 라이브 게임의 구조를 그대로 구현한 멀티플레이 프로젝트.
매치메이킹부터 데디 서버 동적 할당까지 처음부터 끝까지 직접 설계했습니다.
목표
상용 멀티플레이 게임의 서버 구조를 처음부터 끝까지 직접 설계하고 구현
핵심 차별점
로비/배틀 서버 분리 · 포트 동적 할당 · 서버 권위 모델 · 단일 코드베이스
결과물
실제 매치메이킹과 동시 다중 매치가 가능한 동작하는 멀티플레이 게임
02 · AI COLLABORATION

개발 방식 — Harness Engineering

원리를 먼저 이해하고, 그 다음에 에이전트를 활용했습니다.
무엇을 시킬지 알아야 에이전트를 제대로 쓸 수 있기 때문입니다.
왜 이 순서인가
원리를 모르면 에이전트에게 무엇을 시킬지도 모름. 이해가 먼저, 위임이 나중
Harness Engineering
CLAUDE.md로 프로젝트 컨텍스트·규칙을 명시해 에이전트가 일관된 코드 생성
결과
1인 개발로 이 규모의 프로젝트를 완성. AI를 도구로 활용하는 능력 자체도 현대 개발 역량
03 · ARCHITECTURE

시스템 아키텍처

3-tier 구조로 클라이언트 · 백엔드 API · 게임 서버를 분리.
영속 데이터는 HTTP REST로, 실시간 동기화는 KCP/UDP로 — 트래픽 특성에 맞춰 프로토콜을 나눴습니다.
클라이언트
Unity 단일 빌드로 PC·모바일 동시 지원. Mirror로 게임 서버 연결
백엔드 API
계정·인벤토리·가챠·매치 결과 저장. POST /match/acquire로 포트 발급
게임 서버
로비 1개 + 배틀 3개. 모두 Unity 배치모드로 실행되는 독립 프로세스
04 · DESIGN DECISION

동적 포트 할당

배틀 서버를 포트 풀(7778~7780)로 관리해 최대 3개 매치를 동시 진행.
백엔드가 사용 가능한 포트를 발급하고 매치 종료 시 자동 반환합니다.
발급 API
POST /match/acquire → { port: 7779 }
반환 API
POST /match/release — 매치 종료 시 자동 호출
확장 방법
유저 증가 시 풀 크기만 늘리면 됨. 코드 변경 없이 수평 확장 가능
05 · SCENE FLOW

게임 흐름

유저는 5개 씬을 거치며, 각 씬은 독립된 NetworkManager 컨텍스트를 가집니다.
로비 ↔ 배틀 전환 시 StopClient → 새 서버 StartClient로 연결을 재맺습니다.
씬 분리 의도
각 씬이 한 가지 책임만. 로비는 매칭만, 배틀은 전투만 — 디버깅 용이
네트워크 컨텍스트
씬 전환마다 연결을 끊고 새로 맺어 각 씬의 네트워크 상태 독립 유지
메모리 효율
씬 전환 시 이전 씬 리소스 해제. 배틀 중에는 로비 UI가 메모리에 없음
06 · COMMUNICATION

통신 시퀀스

매칭 시점에 클라이언트 ↔ 3종 서버(로비/백엔드/배틀) 사이에서 9번의 메시지가 오갑니다.
백엔드의 포트 발급이 핵심 — 이 단계가 동적 할당의 시작점입니다.
HTTP (실선)
백엔드 API 요청/응답. 로그인 · 매치 포트 발급
Mirror Cmd/Rpc
로비 서버와의 매칭 · 서버 전환 명령
배틀 연결
매치 포트로 4명 동시 접속. 스폰 후 SyncVar로 상태 자동 동기화
07 · SERVER AUTHORITY

서버 권위 모델

모든 게임 로직은 서버에서만 판정. 클라이언트는 입력 전달과 시각화만 담당.
StatUtils.Calculate 단일 진입점으로 클라/서버 동일 로직 보장 — 치트 차단의 핵심입니다.
단일 진입점
StatUtils.Calculate를 클라/서버가 공유. 입력이 같으면 결과도 같음
검증 시점
접속 시 MyAuthenticator가 스탯 재계산 → 불일치 시 연결 거부
동기화 방식
Mirror SyncVar — 서버에서 값 변경 시 모든 클라이언트에 자동 전파
08 · COMBAT SYSTEM

전투 시스템

각 캐릭터는 3개 스킬 슬롯 보유. SkillData는 List<SkillAction>의 조합으로 구성되어
"Dash → Melee" 같은 복합 액션을 자유롭게 만들 수 있습니다.
스킬 조합
SkillData(cooldown + List<SkillAction>)로 복합 액션 구성. 예: Dash → Melee
이중 검증 효과
클라 쿨다운만 있으면 조작 가능. 서버 쿨다운으로 실제 발동 차단
입력 버퍼
0.15초 보존으로 쿨다운 직전 입력도 발동. 체감 반응성 크게 향상
09 · MATCH RESULT

전투 판정 & 결과

180초 동안 팀 전멸 또는 HP 비율 합산으로 승패 결정.
단순 생존자 수 비교가 아닌 HP 비율 방식으로 운빨 요소를 줄였습니다.
HP 비율 판정
팀 HP 비율 합산이 더 공정. 마지막까지 전략적으로 싸울 이유를 만듦
집계 데이터
킬·데스·딜량을 서버가 실시간 추적. ExpCalculator로 경험치 산출
자원 회수
매치 종료 → 포트 반환 → 다음 매치가 즉시 해당 포트 사용 가능
10 · DATA SEPARATION

캐릭터 데이터 구조

ScriptableObject를 로비용 / 전투용 두 가지로 분리.
전투 데이터 변경이 로비 UI에 영향 없고, 컴파일 없이 밸런싱 작업이 가능합니다.
로비/UI 측 SO
자주 안 바뀜. 캐릭터 카드, 인벤토리, 스탯 표시에 사용
전투 메카닉 SO
자주 바뀜. 발사체·데미지·쿨다운 등 밸런싱 시 수정
분리 효과
전투 SO 수정 시 로비 UI 코드 영향 0. 컴파일 없이 밸런싱 가능
11 · INVENTORY

인벤토리 & 강화

드래그&드롭 장비 장착, 강화, 초월 기능 제공.
장비 변경 시 StatUtils.Calculate가 즉시 재계산되어 UI 스탯이 실시간 갱신됩니다.
실시간 반영
장착 변경 시 StatUtils 재계산 → UI 즉시 갱신. 서버 통신 없이 미리보기
서버 동기화
강화/초월 확정 시 백엔드 API 호출 → DB 반영 → 결과 동기화
데이터 모델
PlayerInventory가 런타임 진실. CharacterUIModel이 UI에 바인딩
12 · GACHA

가챠 시스템

로직 · 연출 · UI를 명확히 분리한 4-컴포넌트 구조.
Unity Timeline으로 연출을 제어해 연출 변경이 로직에 영향을 주지 않습니다.
데이터
SO 기반 보상 테이블. 신규 보상 추가 시 코드 수정 없이 에디터에서 작업
로직
확률 계산과 보상 산출이 단일 클래스에 집중. 테스트 용이
연출
Unity Timeline 사용. 디자이너가 코드 없이 연출 시퀀스 수정 가능
13 · PROCEDURAL MAP

맵 생성 시스템

바닥 · 벽 · 덤불을 각각 별도 Generator가 절차적으로 생성.
덤불 안에 들어간 플레이어는 적에게 보이지 않는 은신 시스템이 작동합니다.
바닥 · 벽
절차적 생성으로 다양한 맵 구성 가능. 콜라이더로 충돌 처리
덤불 은신
BushZone + PlayerBushState — 전략적 위치 선점 요소
스폰
팀별 스폰 포인트 · 매치 후 ResetSpawnCounts()로 포트 재사용
14 · CROSS PLATFORM

크로스 플랫폼

IPlayerInputProvider 인터페이스로 입력을 추상화.
PlayerController는 플랫폼을 모름 — 플랫폼별 분기 코드가 한 줄도 없습니다.
설계 원칙
DIP(Dependency Inversion Principle) — 구체가 아닌 추상에 의존
유지보수 효과
한 플랫폼 변경이 다른 플랫폼에 영향 없음. 회귀 버그 위험 차단
확장성
새 플랫폼(콘솔 등) 추가 시 인터페이스 구현만으로 대응 가능
15 · TROUBLESHOOTING

트러블슈팅

기술을 쓴다는 것과 원리를 이해한다는 것은 다릅니다.
아래 4가지 사례는 내부 동작 원리를 직접 파악하고 구조적으로 해결한 경험입니다.
01
투사체 터널링
🤖 Claude Code 에이전트
증상
빠른 투사체가 벽 통과. 디버그 로그로 OnTriggerEnter가 아예 호출 안 됨을 확인
원인 & 해결
Claude Code 에이전트가 코드베이스 분석 → 이산 감지 한계 파악 → SphereCast로 전체 경로 스캔
결과
어떤 속도에서도 벽 관통 없음. 반사 투사체의 충돌 위치도 정확히 계산
02
SyncVar 타이밍 불일치
💬 AI 채팅 + 언리얼 경험
증상
스폰 직후 닉네임·팀 색상이 빈값. 로그로 syncNickname이 빈 문자열임을 확인
원인 & 해결
언리얼 OnRep_ 경험으로 Mirror hook의 필요성을 바로 파악. 값이 클라에 실제 반영되는 순간을 hook으로 잡음
결과
스폰 타이밍 문제 완전 해결. 두 엔진의 동기화 콜백이 같은 원리임을 확인
03
Authenticator 수신 불가
💬 AI 채팅
증상
클라이언트가 인증 메시지를 전송해도 서버 OnAuthRequestMessage가 호출되지 않음
원인 & 해결
인증 완료 전엔 메시지 수신 자체가 차단되는 순환 구조. requireAuthentication = false 옵션 하나로 해결
결과
인증 전 단계에서도 메시지 수신 가능. 이후 인증 흐름 정상 동작
04
로딩바 80% 멈춤
💬 AI 채팅
증상
로딩바 80%에서 갑자기 배틀씬 전환. 로그로 onlineScene에 의한 강제 전환 확인
원인 & 해결
Mirror가 서버·클라 씬 불일치 시 자동 강제 전환한다는 원리 파악. 씬 먼저 로드 → StartClient() 순서로 변경
결과
로딩바 100% 완료 후 씬 전환. 씬 전환과 서버 연결 타이밍을 완전히 제어
