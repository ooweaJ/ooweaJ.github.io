Project Overview

프로젝트 한 줄 소개, 목표와 제약을 정리합니다.

프로젝트 개요

Unreal Engine 5.7 기반 소울라이크 액션 RPG에서, 기존 전투 흐름을 서버 권위 액션 상태 복제 구조로 단계적으로 현대화했습니다. 평타 · 피격 · 장착 아이템 · 보스 스킬 · 사망/부활처럼 결과가 중요한 흐름에서 서버와 클라이언트 결과가 어긋나는 문제를 줄이는 것이 목표입니다.

핵심 원칙 4가지

01 · 서버가 확정

입력은 클라이언트에서 시작할 수 있지만, 최종 액션 상태는 서버가 확정한다.

02 · 태그 기준 재구성

클라이언트는 서버가 복제한 액션 태그를 기준으로 같은 DataTable 액션 데이터를 재구성한다.

03 · 예측 + Skip Owner

소유 클라이언트는 즉시 예측 재생하되, 서버 확정 상태와 충돌하지 않도록 owner 복제는 건너뛴다.

04 · 결과는 서버 권위

데미지 · 사망 · 부활 · 보스 스킬 판정처럼 결과가 중요한 처리는 서버 권위로 제한한다.

Core · Unified Action Component

액션 요청을 하나의 진입점으로 모으고, 예측 재생과 서버 확정을 분리합니다.

7-1통일 액션 컴포넌트

UActionComponent는 평타 경로를 먼저 이전하기 위한 최소 액션 레이어입니다. 액션 요청을 하나의 진입점으로 모으고, 소유 클라이언트 예측과 서버 확정을 분리합니다.

  • TryAction(FGameplayTag) — 액션 요청 단일 진입점
  • 서버는 CommitAction()으로 검증과 실행을 처리
  • 소유 클라이언트는 CanStartAction() 통과 시 즉시 PlayActionLocal() 실행 → 입력 지연 감소
  • 서버 확정 액션은 ActiveAction으로 복제
  • COND_SkipOwner 복제로 예측 재생한 소유 클라이언트의 중복 재생 회피
  • Sequence 증가 → 같은 ActionTag 반복에도 OnRep 재실행 보장

Core · State & Item Replication

상태 변경 권한을 서버로 모으고, 액션의 의미만 태그로 복제합니다.

7-2상태 복제 기준 정리

UStateComponent::Type은 복제 변수였지만 클라이언트 로컬에서 직접 변경될 여지가 있었습니다. 상태 변경 권한을 서버로 모으는 것이 목적입니다.

  • Type을 ReplicatedUsing=OnRep_Type 기준으로 정리
  • ChangeType()은 서버 권위에서만 상태 변경
  • 액션 종료 시 서버는 Idle 복귀, 클라이언트 예측 잠금은 몽타주 종료 기준으로 해제하도록 분리

클라이언트가 임의로 액션 상태를 앞서 변경해 다음 서버 복제와 충돌하는 상황을 줄입니다.

7-3장착 아이템 · 현재 액션 태그 복제

기존 Notify/콤보 구조는 GetCurrentItem()·CurrentData에 크게 의존 — 서버/클라이언트가 다른 데이터를 보면 같은 몽타주라도 판정·콤보가 어긋났습니다.

  • 기본 아이템은 서버에서만 스폰
  • SelectItem → ReplicatedUsing=OnRep_SelectItem 전환
  • AItem을 복제 Actor로 전환 · bAlwaysRelevant·NetUpdateFrequency 조정
  • CurrentActionTag 복제 → OnRep에서 로컬 DataTable 재조회로 CurrentData 재구성

최소 복제 선택: FActionData 전체가 아니라 의미를 나타내는 GameplayTag만 복제해 네트워크 상태를 작게 유지합니다.

  • AGruxMeteor를 복제 Actor로 전환 · 이동 복제 ON
  • Target · Damage 등 메테오 상태를 복제 속성으로 추가
  • 타이머 시작 · 투사체 활성화 · 충돌 · 데미지 · 파괴는 서버 권위에서만 실행

  • 폭발 이펙트는 서버 Multicast 전파, Actor 삭제는 짧게 지연해 이펙트 RPC 유실 방지

사망 완료가 Anim Notify에만 의존하면 Notify 누락 시 부활 타이머가 시작되지 않을 수 있었고, 서버 권위 상태 복구와 클라이언트 표현 복구가 한 함수에 섞여 있었습니다.

  • 사망 완료 처리는 서버에서 한 번만 예약
  • 사망 Multicast는 visibility · 사망 이펙트 같은 표현만 처리
  • 3초 후 서버 타이머가 부활 Multicast 호출
  • HP 회복 · State->SetIdleMode()는 authority 인스턴스에서만
  • 입력 · visibility · 몽타주 정리 · 이동 복구는 각 클라이언트 표현에 맞게
  • GameMode 리스폰 타이머는 함수 로컬 핸들로 사용해 다중 플레이어 타이머 덮어쓰기 방지

1소유 클라이언트 평타 입력 지연

문제
기존 평타는 Server RPC → Reliable Multicast 이후에 몽타주가 재생되어, 지연 환경에서 소유 클라이언트 입력 반응이 늦어질 수 있었다.
원인
소유 클라이언트도 서버 응답을 기다린 뒤 같은 Multicast 경로로 표현을 받는 구조였다.
해결
TryAction()에서 소유 클라이언트가 조건을 통과하면 즉시 로컬 예측 재생, 서버에는 Server_TryAction(). 서버 확정 액션은 ActiveAction으로 복제하되 owner엔 COND_SkipOwner 적용.
결과
평타 경로에서 입력 즉시 반응과 서버 권위 액션 확정을 동시에 두는 기준선이 생겼다.

2장착 아이템 / 현재 액션 데이터 불일치

문제
Anim Notify·콤보 로직이 현재 아이템·액션 데이터에 의존하지만, 서버와 클라이언트가 같은 데이터를 본다는 보장이 부족했다.
원인
장착 아이템 참조와 CurrentData 전체가 네트워크 기준으로 안정화되어 있지 않았다.
해결
선택 아이템 Actor를 서버 스폰/복제 구조로 정리하고, 현재 액션은 FActionData 전체 대신 CurrentActionTag를 복제해 클라이언트가 DataTable에서 같은 데이터를 재구성하게 했다.
결과
액션 의미를 태그로 맞추는 최소 복제 구조가 생겼고, 이후 콤보 인덱스·섹션·시작 시간 확장 기반이 마련됐다.

3보스 메테오 이펙트와 판정 불일치

문제
Grux 메테오가 클라이언트에서 안정적으로 보이지 않거나, 충돌 직후 Actor가 삭제되어 폭발 이펙트 Multicast를 놓칠 수 있었다.
원인
메테오 Actor 복제와 서버 권위 충돌 처리가 부족했고, Multicast 직후 즉시 Destroy()가 호출되는 구조였다.
해결
메테오를 복제 Actor로 전환하고 서버에서만 충돌/데미지/파괴 처리. 이펙트 Multicast 후 충돌·이동을 끄고 SetLifeSpan(1.f)로 지연 삭제.
결과
보스 스킬의 실제 판정은 서버가 결정하고, 클라이언트는 복제 Actor와 Multicast 이펙트를 관찰하는 구조로 정리됐다.

4사망 후 부활 / 입력 복구 불안정

문제
사망 Notify 누락·타이밍 어긋남 시 부활 타이머가 시작되지 않거나, 부활 후 입력·이동 가능 상태가 복구되지 않을 수 있었다.
원인
사망 완료·부활 타이머·HP/State 복구·visibility/input 복구가 같은 흐름에 섞여 있었고 일부가 클라이언트에서도 직접 실행될 수 있었다.
해결
서버가 사망 완료와 부활 타이머를 한 번만 관리하고, 클라이언트는 Multicast로 표현 상태를 복구하도록 분리. 부활 시 Status->SetMove()로 이동 가능 상태 복원.
결과
사망/부활의 서버 권위 상태와 클라이언트 표현 책임이 더 명확해졌다.
